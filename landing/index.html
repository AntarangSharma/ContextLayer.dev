Push, not just pull

context_validate — verdicts, not suggestions.

Most retrieval is pull: the agent has to know to ask. ContextLayer also pushes — the agent shows its proposed change and gets back a verdict with violated rules, severity, and citations.

FLOW
  1. 1
    Agent drafts a change
    Snippet, file path, intent.
  2. 2
    Calls context_validate
    Hybrid retrieval matches the proposal against rule atoms.
  3. 3
    Gets back a verdict
    Passes? Violations? With severity + source PR/commit.
  4. 4
    Rewrites — or proceeds
    Convention enforcement, not suggestion.

Pipeline

Three models. One pipeline. ~$1.50/repo.

Cheap models do the heavy lifting; the expensive one only sees what matters.

STAGE 1 ~$0.15
🔍

Haiku · Filter

STAGE 2 ~$0.30
⚗️

Sonnet · Extract

batched 15/call, cached

Opus · Structure

1 call · extended thinking

Dedup, topic clustering, rule promotion. Yields ~40 canonical atoms ready for retrieval.

Drop-in

One block of JSON. You're done.

Add this to your project's .mcp.json.

{
  "mcpServers": {
    "contextlayer": {
      "command": "uvx",
      "args": ["contextlayer-dev", "mcp", "--repo", "."]
    }
  }
}
vs. the rest

Why this is different.

Most "AI context" tools treat the repo as a text corpus. ContextLayer treats it as a history of decisions.

What others do What ContextLayer does
Stuff the whole repo into every prompt Extract ~40 canonical atoms that capture intent
Vector-search over raw file chunks Hybrid retrieval over structured atoms with citations
Hope the agent reads CLAUDE.md Push verdicts via context_validate — agent can't ignore it
SaaS dashboard, vendor lock-in Local SQLite, MIT license, one Python package
Static rule files that go stale Re-indexed from live git history + decision journal
One model fits all Haiku → Sonnet → Opus, right model per stage
